# N4 — 用户可测入口与诊断壳

## 任务类型
执行入口设计 / 诊断壳 / 使用说明

## 目标
让用户第二天不用读大量代码，就能直接测试 Proto-Self Kernel v1 当前效果。

## 当前层级
用户可体验收口层

## 当前确定项
- 最小 Telegram 主链已验证
- 需要把可体验入口搭出来

## 关键未知
- 当前是否已有足够清晰的只读诊断入口
- 用户是否能快速看懂“预期现象 vs 异常现象”
- 测试手册是否能避免误判

## 成功判据
1. 有一套用户第二天可直接执行的测试说明
2. 有一组固定测试场景：
   - 重复相似事件
   - 失败回流事件
   - 可区分意图事件
3. 有只读诊断入口或等价脚本，至少能看到：
   - 最近 trace 关键字段
   - cycle hits / cycle id
   - reflection_trigger
   - revision_counter 变化
4. 有一份“预期现象 / 异常现象 / 排查入口”对照表
5. 更新 `reports/N4_REPORT.md`
6. 更新 `runtime/RUN_STATE.json`

## 硬约束
- 不要把调试壳设计成能越权改状态
- 不要把主观感受当唯一验收
- 不要要求用户读很多内部文件才能测试
- 不要让用户测试依赖不稳定样本

## 允许修改
- 只读诊断命令
- 测试脚本
- 使用说明
- 对照表
- 必要的轻量壳层

## 禁止修改
- 越权写状态
- 绕过 EgoCore 做直接对外执行
- 无证据更新正式状态页

## 必交付物
- 用户测试说明
- 测试场景脚本或清单
- 诊断入口或脚本
- `reports/N4_REPORT.md`
- `runtime/RUN_STATE.json` 更新

## 必须记录的证据
- 实际诊断输出示例
- 用户测试步骤
- 每一步预期看到的现象
- 若只实现壳未做真实测试，必须标记 `implemented_but_pending_real_validation`

## 失败出口
若当前主链还不足以支撑用户自测：
- 不要硬交互壳
- 报告中明确“缺哪一个观测字段/入口”
- 给出最小补齐动作
