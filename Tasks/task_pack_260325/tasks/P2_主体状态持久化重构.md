# P2：主体状态持久化重构任务单

## 任务编号
P2

## 任务类型
状态架构 / 持久化 / 主体连续性收口

## 目标
把当前可能过于粗糙的 mirror/state.json 方案重构为清晰的状态层次与持久化设计，明确 agent-global / session / thread / experiment 的边界，避免把主体连续性做成单文件偶然产物。

## 成功判据
1. 明确长期主体状态与会话/线程/实验态分层
2. `/new`、session reset、restore、replay 对应的状态影响面有清晰定义
3. 不再只依赖单个裸 `state.json` 承担所有主体连续性语义
4. 有迁移策略，不破坏已有状态或样本

## 当前层级
架构层 / 状态主权层

## 当前确定项
- 当前 adapter 默认读写单个 `artifacts/proto_self_mirror/state.json`
- 这很容易把长期状态和局部运行态混在一起

## 关键未知
- 现有 `/new` 与 reset 实际应该保留哪些主体信息
- 真正需要几层状态隔离
- replay / compare / experiment 是否已有潜在需求但尚未正式编码

## 为什么这个任务优先级高
如果状态主权不收口，后面所有连续身份、记忆演化、观察期都会被单文件设计拖垮。

## 强制范围
- ProtoSelfAdapter state load/save
- Proto-Self state mirror 方案
- restore / replay / reset 相关路径
- 必要的状态 schema 版本与迁移设计

## 明确不做
- 不在此任务里扩写更多 identity/appraisal 本体逻辑
- 不把 OpenEmotion 本体逻辑搬回 EgoCore

## 开发前六问（必须先答）
### A. Capability Ownership
主体本体语义归 OpenEmotion；host-side state store / mirror / restore orchestration 归 EgoCore。

### B. Authority Source
字段语义权威源在 OpenEmotion；宿主持久化边界与 reset 语义归 EgoCore。

### C. Mirror Need
允许 host-side mirror，但不得让 mirror 取代 OpenEmotion 本体真相源。

### D. Boundary Risk
若 EgoCore 单文件 state 事实上定义了主体语义，就是双主风险。

### E. Failure Owner
状态恢复与 reset 出问题由 EgoCore 兜底；本体解释仍由 OpenEmotion 负责。

### F. Exit Plan
临时 state layout 必须带版本号与迁移说明；最终保留最薄 host-side store。

## 主执行链（只保留 1 条）
1. 盘点当前所有状态读写点
2. 区分长期主体态 / 会话态 / 运行态 / 实验态
3. 设计最小状态目录与 schema version
4. 实现兼容加载与迁移
5. 补 replay / reset / restore 测试

## 备选链（仅当主链被证伪时启用）
若完整分层一次做不完，先把单文件 state 拆成长期态 + 会话态两层，并明确 `/new` 语义。

## 强制产出
- artifacts/P2/TASK_REPORT.md
- artifacts/P2/STATE_OWNERSHIP_MAP.md
- artifacts/P2/MIGRATION_PLAN.md
- 新状态目录 / store 实现
- 回归与迁移测试

## 强制证据
- 状态读写点清单
- 新旧状态图
- reset / restore / replay 行为样本
- 迁移前后兼容证据

## 验收口径限制
只能报‘状态持久化边界已重构到某层级’，不得直接报‘长期连续身份已成立’。

## 失败后回退动作
若迁移风险未控，保留兼容读取并延后删除旧 state 文件。

## 交给 Codex 的硬要求
- 先画状态层级图
- 不得偷把主体语义塞回宿主
- 必须给 `/new`、reset、restore、replay 的语义表
- 没有迁移计划不得动正式路径
