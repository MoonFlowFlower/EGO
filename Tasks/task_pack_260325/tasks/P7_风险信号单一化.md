# P7：风险信号单一化任务单

## 任务编号
P7

## 任务类型
规则收口 / 风险治理 / 信号权威源统一

## 目标
将 risk signal 的权威源、字段名、打分器、trace 映射和测试矩阵统一，消除多处手抄词表、字段别名和语义不一致的问题。

## 成功判据
1. 风险信号只有一个权威源
2. 字段名、词表/规则、trace、evidence 统一
3. 高/中/低风险路径在 runtime 与 proto-self 中行为一致
4. 有可持续维护的测试矩阵

## 当前层级
规则治理层 / 边界一致性层

## 当前确定项
- 当前 loop.py 中存在风险关键词硬编码，并标注要与其他模块保持一致
- 已出现 risk/risk_level 兼容痕迹

## 关键未知
- 还有哪些模块维护自己的风险规则副本
- 当前高风险/低风险样本覆盖度是否足够

## 为什么这个任务优先级高
风险信号一旦分裂，最容易导致真实主链表现和证据解释互相打架。

## 强制范围
- risk keyword / scorer / field mapping / trace / evidence / tests
- 与 Proto-Self、Runtime、Telegram evidence 直接相关的风险路径

## 明确不做
- 不扩成完整安全系统重构
- 不把简单风险规则过度产品化

## 开发前六问（必须先答）
### A. Capability Ownership
高风险动作裁决归 EgoCore；主体接收的 risk signal 映射要与宿主一致。

### B. Authority Source
risk canonical source 在 EgoCore 风险模块；OpenEmotion 只消费标准化 risk signal。

### C. Mirror Need
允许适配映射，不允许多个模块各自维护真词表。

### D. Boundary Risk
多个词表/字段共存会形成风险语义双主。

### E. Failure Owner
EgoCore 安全边界层兜底。

### F. Exit Plan
统一后删除手抄副本并补测试保护。

## 主执行链（只保留 1 条）
1. 盘点所有 risk 相关字段与规则
2. 指定 canonical risk source
3. 收口 mapping 与字段
4. 建高/中/低风险样本矩阵
5. 跑回归并更新 evidence

## 备选链（仅当主链被证伪时启用）
若一次性统一不了，先统一 canonical field + 单一 scorer，再逐步去掉旧副本。

## 强制产出
- artifacts/archive/repo_cleanup_history/P7/TASK_REPORT.md
- artifacts/archive/repo_cleanup_history/P7/RISK_SIGNAL_MAP.md
- canonical risk module
- 测试矩阵
- 文档更新

## 强制证据
- 词表/规则盘点
- 样本矩阵
- 行为对照
- does_not_prove 字段

## 验收口径限制
只能报‘风险信号已统一到某层级’，不得报‘安全已完全收口’。

## 失败后回退动作
若统一后误报升高，回退到单 scorer + 兼容 mapping，而不是恢复多副本。

## 交给 Codex 的硬要求
- 先盘点再统一
- 一个风险字段只能有一个 canonical source
- 必须保留样本矩阵
- 不准让词表散落复制
