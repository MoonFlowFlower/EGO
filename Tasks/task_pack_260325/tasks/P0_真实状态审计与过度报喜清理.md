# P0：真实状态审计与过度报喜清理任务单

## 任务编号
P0

## 任务类型
审计 / 治理 / 文档-代码-证据对账

## 目标
建立一份唯一可信的程序状态矩阵，识别并清理 README、验收文档、代码现状、artifacts 之间的口径冲突，特别是提前报喜、过度报完成、证据层级越级表述的问题。

## 成功判据
1. 产出唯一状态矩阵：功能项 / 当前层级 / 最高证据 / 主链接入状态 / 启用状态 / 本次不能证明什么
2. 标出所有“文档口径 > 证据层级”的条目
3. 对 README、任务汇报、验收报告提出精确降级或升级建议
4. 给出后续 7 个任务的真实前置条件，而不是沿用模糊表述

## 当前层级
治理层 / 主链可信度层

## 当前确定项
- README 已存在“稳态收口”“真实 Telegram 验证通过”“unified runner 一致性验证”等较强口径
- 验收证据协议明确规定：没有 E5 不得写“关键未知为无 / 稳定性已收口”，没有 E6 不得写“正式长期固化”
- simulated / integration / real_channel 必须严格区分

## 关键未知
- 当前各结论到底最高只配到 E2 / E3 / E4 / E5 的哪一级
- 是否已经存在失败样本账本
- README 中哪些口径已经越级
- 是否有未同步的真实 artifacts 缺口

## 为什么这个任务优先级高
如果这一层不先清，后续所有重构都会建立在错误前提上；你最怕的不是代码差一点，而是系统持续在错误结论上自我强化。

## 强制范围
- README.md
- PROGRAM_STATE / 验收报告 / 任务单 / artifacts 索引
- 与 RuntimeV2Loop、Proto-Self、Telegram 验证直接相关的证据材料
- 所有自称 verified / 已接主链 / 已启用 / 已生效 / 已收口 的表述

## 明确不做
- 不直接改业务逻辑
- 不在没有审计清楚前直接删代码
- 不把“我猜应该是这样”写成事实

## 开发前六问（必须先答）
### A. Capability Ownership
EgoCore 负责对外口径、验收与证据治理；OpenEmotion 只提供其本体输出事实，不负责对外夸口。

### B. Authority Source
证据层级与对外口径的权威源在验收协议与真实 artifacts，不在口头总结。

### C. Mirror Need
允许读取历史总结作为线索，但不得把历史口头结论当真相源。

### D. Boundary Risk
若把 README 或任务总结当正式真相源，会导致文档口径倒逼代码现实，形成伪闭环。

### E. Failure Owner
EgoCore 治理层兜底；发现口径越级必须由宿主侧修正。

### F. Exit Plan
审计完成后，README/文档仅保留与状态矩阵一致的口径；历史过度表述进入 failure evidence。

## 主执行链（只保留 1 条）
1. 收集当前所有强口径表述
2. 为每个表述追对应 artifacts / tests / real samples
3. 依据 E0~E6 协议重新标注最高证据级别
4. 产出统一状态矩阵与降级/升级建议
5. 必要时直接修改 README / 验收模板 / 任务模板

## 备选链（仅当主链被证伪时启用）
若 artifacts 不全，先输出“证据缺口矩阵”，明确哪些条目只能降级为‘已实现/已接候选/待验证’。

## 强制产出
- artifacts/P0/TASK_REPORT.md
- artifacts/P0/STATE_MATRIX.md
- artifacts/P0/WORDING_CONFLICTS.md
- README 修订建议或实际修改
- FAILURE_SAMPLES.md（至少纳入口径越级样本）

## 强制证据
- 每条强口径的原文位置
- 对应真实 artifact / test / log / report
- E0~E6 重新标级结果
- does_not_prove 字段

## 验收口径限制
没有 E5，不得写“稳定收口 / 关键未知为无”；
没有 E6，不得写“正式长期固化 / 稳定依赖前提”；
单次真实样本最多支持“样本级生效”或“已接真实主链”。

## 失败后回退动作
如果审计材料不足，保守降级，不允许维持强口径。

## 交给 Codex 的硬要求
- 先审计，后修改
- 必须给出逐条证据映射，不接受泛泛总结
- 必须显式写出“本次结论不能证明什么”
- 不允许把缺证据解释成“隐含成立”
