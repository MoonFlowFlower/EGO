# 《EGO 验收证据分级协议 v1》

## 0. 一句话结论

**任何“已接主链 / 已启用 / 已生效 / 已闭环 / verified”结论，结论强度不得高于证据层级。**
凡无真实主链证据，不得用真实生效口径；凡无真实触发证据，不得报闭环完成。

---

## 1. 协议目标

本协议用于统一 EGO 项目的验收口径，解决以下问题：

1. 把 **模拟通过** 误报为 **真实主链通过**
2. 把 **局部脚本成功** 误报为 **系统级生效**
3. 把 **建议 / 候选 / 临时方案** 误报为 **正式能力**
4. 把 **无真实触发证据** 的结果误报为 **闭环完成**

本协议适用于：

* EgoCore
* OpenEmotion
* 双核接口
* Telegram / 其他外部通道
* runtime / tool / memory / reflection / replay / gate / CI
* 所有新功能、修复、重构、验收、上线口径

---

## 2. 核心原则

### 2.1 结论强度不得高于证据强度

没有更高层级证据，不得使用更高层级结论。

### 2.2 真实主链优先于模拟链

模拟、单测、回放都只能作为辅证；涉及正式生效时，真实主链证据优先。

### 2.3 主链接入、启用、生效、观察是四个不同层级

不得混用：

* **已实现** ≠ 已接主链
* **已接主链** ≠ 已启用
* **已启用** ≠ 已生效
* **已生效** ≠ 已完成观察期

### 2.4 未接主链、未启用、无真实触发证据，不得报完成

只能使用：

* 已实现但待验证
* 部分完成
* 尚未生效
* 证据不足
* 过渡方案
* 临时止血

### 2.5 模拟证据必须显式标注

所有证据必须标明来源层级，不允许把 simulated / mock / replay 伪装成真实主链。

---

## 3. 验收证据分级

## E0 — 文档 / 设计级证据

**定义**：只有设计、方案、文档、伪代码、接口草案。
**可证明**：

* 思路存在
* 设计已产出
* 接口已定义

**不可证明**：

* 已实现
* 可运行
* 已接主链
* 已启用
* 已生效

**允许口径**：

* 已设计
* 已形成方案
* 已完成文档化

**禁止口径**：

* 已完成
* 已修复
* 已生效
* 已闭环

---

## E1 — 静态 / 局部实现级证据

**定义**：代码存在、模块存在、静态检查通过、局部函数可跑。
**典型证据**：

* 文件已生成
* lint/type check 通过
* schema 存在
* 函数局部运行通过

**可证明**：

* 已实现局部能力
* 具备进入更高验证层的前提

**不可证明**：

* 集成可用
* 主链可用
* 真实渠道可用

**允许口径**：

* 已实现
* 已落代码
* 已具备集成前提

**禁止口径**：

* 已接主链
* 已启用
* 已生效
* verified

---

## E2 — 单元 / 模拟级证据

**定义**：单测、mock、simulated smoke、fake driver、离线 replay。
**典型证据**：

* unit tests
* simulated smoke
* mocked Telegram update
* replay / tape 离线回放
* classifier 离线验证

**可证明**：

* 内部逻辑大体正确
* 明显边界错误被拦截
* 基本脚本级闭环存在

**不可证明**：

* 真实集成稳定
* 真实 Telegram 主链稳定
* 用户侧真实可用

**允许口径**：

* 脚本级通过
* 模拟验证通过
* 最小闭环已实现
* 已具备集成测试条件

**禁止口径**：

* 已接主链
* 已启用
* 已生效
* verified_mainline_e2e

---

## E3 — 集成级证据

**定义**：真实模块之间联调，但外部渠道仍是受控环境。
**典型证据**：

* EgoCore ↔ OpenEmotion 结构化联调
* runtime + tool + replay 联调
* testbot / sandbox channel
* 近真实但受控的端到端链路

**可证明**：

* 多模块真实集成可跑
* 接口基本正确
* 主链候选可进入真实验证

**不可证明**：

* 生产真实渠道稳定
* 真实用户场景可靠
* 长时间观察无问题

**允许口径**：

* 集成验证通过
* 已具备真实主链验证条件
* 主链候选可用

**禁止口径**：

* 已正式生效
* 已稳定运行
* 已完成观察期

---

## E4 — 真实主链触发级证据

**定义**：真实渠道、真实入口、真实交付、真实行为反馈。
**典型证据**：

* 真实 Telegram bot / chat / user message
* 真实 message send / reply / edit / wait / block
* 原始 update + 实际 outbox + timeline
* 真实 session lifecycle
* 真实 runtime orchestration

**可证明**：

* 已接主链
* 已启用
* 在样本范围内已生效

**不可证明**：

* 稳定性已收口
* 长期可靠
* 全场景无问题

**允许口径**：

* 已接入真实主链
* 已启用
* 已有真实触发证据
* 样本级生效

**禁止口径**：

* 关键未知为无
* 完全稳定
* 全面闭环完成

---

## E5 — 观察期 / 稳定运行级证据

**定义**：真实主链已运行一段时间，且有连续样本、失败归因、回归闭环。
**典型证据**：

* 多个真实样本
* 连续若干天/若干任务
* 已覆盖真实失败样本回归
* 稳定性指标/错误率/告警记录

**可证明**：

* 已生效并进入观察期
* 稳定性初步成立
* 可进入更正式的长期阶段

**不可证明**：

* 永久无问题
* 无限泛化
* 不再需要回归

**允许口径**：

* 已生效，观察中
* 初步稳定
* 进入观察期

**禁止口径**：

* 关键未知为无
* 永久完成
* 不再需要验证

---

## E6 — 长期固化级证据

**定义**：真实运行长期稳定，并已纳入固定 gate / CI / regression / governance。
**典型证据**：

* CI 强制门禁
* 主链强制使用
* 长期样本稳定
* 失败闭环成熟
* 文档/代码/流程一致

**可证明**：

* 正式固化
* 长期主链能力成立
* 可作为下游依赖前提

**允许口径**：

* 已固化为正式主链能力
* 已纳入长期治理
* 长期有效

---

## 4. 结论口径映射表

| 结论口径     | 最低所需证据层级      |
| -------- | ------------- |
| 已设计      | E0            |
| 已实现      | E1            |
| 脚本级通过    | E2            |
| 集成验证通过   | E3            |
| 已接主链     | E4            |
| 已启用      | E4            |
| 已生效      | E4 + 明确真实触发样本 |
| 已生效，观察中  | E5            |
| 已固化为正式能力 | E6            |

### 强制规则

* **没有 E4，不得写：已接主链 / 已启用 / 已生效**
* **没有 E5，不得写：关键未知为无 / 稳定性已收口**
* **没有 E6，不得写：正式长期固化 / 可作为稳定依赖**

---

## 5. 每份验收必须包含的字段

所有验收报告必须固定包含以下字段：

### 5.1 基本字段

* 任务名称
* 当前层级
* 证据层级
* 主链接入状态
* 启用状态
* 结论口径
* 当前确定项
* 关键未知
* 本次结论不能证明什么
* 下一步最小闭环动作

### 5.2 证据清单字段

每条证据必须有：

* `evidence_id`
* `evidence_level`（E0~E6）
* `source_type`（doc / unit / simulated / integration / real_channel / observation）
* `artifact_path`
* `what_it_proves`
* `what_it_does_not_prove`

没有 `what_it_does_not_prove` 的证据，视为不合格证据。

---

## 6. 真实渠道类能力的额外强制规则

凡涉及 Telegram、邮件、日历、工具执行、真实用户交互等外部主链，必须额外满足：

### 6.1 必须区分三类测试

* `simulated`
* `integration`
* `real_channel`

### 6.2 最终口径必须以 `real_channel` 为准

如果 simulated / integration 通过，但 real_channel 未通过：

> 结论只能是：
> **已实现或已通过模拟/集成验证，真实主链未通过或待验证**

### 6.3 真实 Telegram 最小证据包

至少包含：

* 原始 Telegram update
* 归一化 event
* OpenEmotion 结构化结果
* EgoCore response plan
* 实际发送记录
* timeline / tape / replay artifact

少任一项，不得报真实主链通过。

---

## 7. 失败证据协议

所有失败都必须成为正式证据，不允许只保留成功样本。

### 每个失败样本至少包含

* 失败编号
* 触发方式
* 输入
* 预期
* 实际
* 所属证据层级
* 初步归因
* 是否已纳入回归
* 修复后是否复测

### 失败归因分类

必须落入以下之一：

* boundary_error
* authority_source_error
* contract_error
* schema_error
* runtime_error
* orchestration_error
* delivery_error
* state_recovery_error
* replay_mismatch
* test_gap
* wording_overclaim

---

## 8. 禁止事项

### 禁止 1

把模拟 smoke / replay / unit test 结果写成“真实主链生效”。

### 禁止 2

把“文件已存在 / 代码已生成”写成“功能已完成”。

### 禁止 3

把“已接入口”写成“已启用”。

### 禁止 4

把“已启用”写成“已生效”。

### 禁止 5

把“单次真实样本通过”写成“稳定无未知”。

### 禁止 6

把“观察期建议”与“已完全收口”同时写在同一报告里而不解释冲突。

## 9. 标准汇报模板

下面这份模板可直接复用：

```md
# 验收汇报

## 任务名称
[任务名]

## 当前层级
[设计级 / 实现级 / 脚本级 / 集成级 / 真实主链级 / 观察级 / 固化级]

## 证据层级
[最高只写到当前真实拥有的层级，例如 E2 / E3 / E4]

## 主链接入状态
[未接入 / 已接入候选 / 已接入真实主链]

## 启用状态
[未启用 / 已启用候选 / 已启用]

## 结论口径
[已设计 / 已实现 / 脚本级通过 / 集成验证通过 / 已接主链 / 已生效，观察中...]

## 真实触发证据
- [证据 1]
- [证据 2]

## 当前确定项
- [确定项 1]
- [确定项 2]

## 关键未知
- [未知 1]
- [未知 2]

## 本次结论不能证明什么
- [不能证明 1]
- [不能证明 2]

## 证据清单
| evidence_id | level | source_type | artifact_path | proves | does_not_prove |
|---|---|---|---|---|---|

## 下一步最小闭环动作
[下一步]
```

## 10. 你给施工体的固定执行令

以后你可以把下面这段直接贴给 Claude Code / OpenClaw：

```md
按《EGO 验收证据分级协议 v1》执行。

强制要求：
1. 所有结论强度不得高于证据层级
2. 所有证据必须标注 E0~E6 与 source_type
3. 没有 real_channel / real_telegram 证据，不得使用“已接主链 / 已启用 / 已生效 / verified”口径
4. 必须显式写出“本次结论不能证明什么”
5. 必须同时提供成功样本与失败样本
6. 若存在观察期建议，不得同时报“关键未知为无”或“已完全收口”
```

## 11. 一句话收口

> **EGO 的验收，不看“它说自己过了没有”，只看“它拿出了哪一层证据，因此最多配得上哪一层结论”。**
