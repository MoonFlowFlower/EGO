# Legacy Algorithm Inventory

This inventory records reusable ideas from the archived pre-operator `EgoCore`, `OpenEmotion`, and `ego_desktop_lab` surfaces.

Boundary:

- Reference only.
- No runtime authority.
- No default path.
- No fallback runtime.
- No task-routing authority.
- Reuse requires a new Stage Card and evidence gate.

Archive pointer: `legacy-pre-operator-mainline-before-purge`

## Self-Model Snapshot

Reusable idea: maintain a bounded snapshot of identity, preferences, constraints, and recent continuity signals so later reasoning can inspect an explicit self-model candidate.

Reuse boundary: this may inspire future `EgoOperator` memory or identity-continuity work, but only as a proposal. It must not bypass operator approval, memory promotion gates, or trace requirements.

## Appraisal Candidate

Reusable idea: compute candidate appraisal signals from context, goals, uncertainty, and risk before selecting a response or action.

Reuse boundary: any future appraisal signal must remain advisory until admitted by a new runtime contract. It must not become a hidden behavior driver or a replacement for LLM understanding plus runtime gate.

## Reflection Support

Reusable idea: generate structured reflection notes, revision candidates, or explanation traces after an event so the system can compare intended behavior against observed outcome.

Reuse boundary: reflection outputs are diagnostic/proposal artifacts only unless a new Stage Card defines promotion rules, evidence expectations, and rollback behavior.

## Initiative Proposal

Reusable idea: formulate bounded initiative candidates such as follow-up suggestions, reminders, or proactive topics while separating proposal from execution.

Reuse boundary: initiative cannot trigger real side effects, transport, memory mutation, or default proactive behavior without explicit `EgoOperator` gate ownership and evidence.

## Telegram / Proactive Historical Evidence

Reusable idea: historical Telegram and proactive runs provide comparison cases for visible-text quality, transport boundaries, askback regressions, and gate discipline.

Reuse boundary: these records are historical evidence only. They do not authorize old Telegram paths, old proactive main-chain behavior, or old runtime ownership in the current main branch.

## Required Reuse Gate

Any reuse must document:

- Problem reframe.
- One hypothesis.
- One change surface.
- Authority source.
- What can change.
- What cannot be proven.
- Three-level verification.
- Rollback plan.
- Claim ceiling.

No inventory item proves stable user benefit, live autonomy, durable memory efficacy, functional selfhood, consciousness, or subjective experience.
