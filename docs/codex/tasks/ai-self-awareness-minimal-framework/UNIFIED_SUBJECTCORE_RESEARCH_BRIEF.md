# Unified SubjectCore Research Brief

> Planning-only supplement inside the closed `ai_self_awareness_research` lane.
> This file is not a new authority source. Current repo authority remains `docs/PROGRAM_STATE_UNIFIED.yaml`.

## Why this supplement exists

Selection closeout already fixed `active-inference self-model` as the durable build-first candidate under the current bounded research gate. That result does **not** answer a different product/research question:

- how to make the system feel like one continuous subject
- how to keep that continuity without collapsing into narrative-only optimization
- how to add proactive proposal capability without widening runtime authority

This supplement freezes a post-closeout framing for that question without reopening candidate competition, changing the current execution owner, or widening the host-consumable surface.

The initial A/B/C compare has now already done its job:

- it is a completed bounded architecture reading
- it is not a continuing admission race

## Core judgment

The repo should **not** regress to `memory-only`, and it also should **not** jump to neuron-level brain simulation.

The recommended framing is:

- **externally unified object**
  - one `SubjectCore` facade that presents one self-thread
- **internally layered mechanisms**
  - memory/projection for continuity
  - state/writeback for plasticity
  - proposal/governor bridge for bounded initiative

This keeps the user-facing subject coherent while preserving the mechanism separation required for falsifiability, replayability, and host governance.

## Why memory-only is insufficient

`memory-only` is strong at:

- identity continuity
- same-session recall
- profile persistence
- human-readable continuity cues

`memory-only` is weak at:

- stable failure-dependent policy change
- tension/viability changing later action tendency
- proposal discipline under host governance
- causal replay of why behavior changed

Current repo evidence already points the same way:

- formal mainline keeps `episodic_trace / corrective_trace / writeback` on the live `proto_self` path
- standalone `openemotion.memory/*` lines are currently not on the formal mainline
- the smallest surviving candidate in prior synthetic work was not generic memory, but `compact self-state + boundary + counterfactual + writeback`

## Why neuron-level framing is not the next build step

This repo does not currently need low-level neural simulation to answer the active engineering question.

The right abstraction level is **mid-level computational motifs**:

- hippocampus/PFC-style memory-planning coupling
- predictive / active-inference state updating
- bounded workspace / governor gating
- replay / rollout / counterfactual correction loops

That level is:

- implementable in the current stack
- falsifiable with replay and controlled observation
- compatible with the existing host-governed boundary

## Unified SubjectCore framing

The planning target is one facade:

- `SubjectCore.identity_continuity`
- `SubjectCore.self_state`
- `SubjectCore.memory_projection`
- `SubjectCore.proposal_engine`
- `SubjectCore.governor_bridge`

The facade is **object-level**.
The internals remain **mechanism-level**.

This is the recommended answer to the “面向对象 vs 面向过程” concern: the repo should expose one coherent subject object while still letting the implementation use layered recursive loops.

## The four mid-level primitives to study

### 1. Identity continuity

- `self-summary + profile + session recall`
- purpose: keep one self-thread legible to the model and the user

### 2. Plasticity

- `counterfactual writeback + corrective trace`
- purpose: make failed or blocked outcomes change later tendency

### 3. Tension control

- `viability / uncertainty / calibration / repair bias`
- purpose: let internal pressure causally shift later behavior

### 4. Bounded initiative

- `rollout / replay / proposal loop`
- purpose: let the subject propose thinking/search/message actions before execution authority is granted

## A/B/C compare as completed architecture reading

### A. Memory-only

- continuity-first control arm
- expected to score well on readability and self-thread continuity
- expected to underperform on plasticity and bounded initiative

### B. State-only-minimal

- behavior-first control arm
- expected to preserve corrective traces and tendency change
- expected to underperform on readable identity continuity

### C. Hybrid unified-core

- target candidate shape
- one `SubjectCore` facade
- internal coordination across memory + state + proposal/governor
- expected to be the only arm that can jointly preserve continuity, plasticity, and bounded initiative
