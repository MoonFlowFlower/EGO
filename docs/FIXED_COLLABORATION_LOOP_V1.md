# Fixed Collaboration Loop v1

> 适用范围：EGO repo 内由用户 + Codex 持续推进的研发 /研究 / bounded runtime 改动。
> 这是协作执行契约，不替代 `docs/PROGRAM_STATE_UNIFIED.yaml`。

## 目标

把“每轮重新讨论怎么协作”收敛成一个固定闭环：

- 用户负责：方向、风险边界、是否升级 gate
- Codex 负责：问题重构、最小实现、验证、记账、结论口径

目的不是减少真实判断，而是减少低价值往返，让项目能稳定推进到更强的：

- `continuity`
- `plasticity`
- `proposal-only initiative`

同时不把 bounded evidence 误报成：

- cross-entry proof
- runtime efficacy
- autonomy expansion
- consciousness claim

## 当前 authority 前提

- 正式 authority source 仍是 `docs/PROGRAM_STATE_UNIFIED.yaml`；本文件只定义协作节奏，不提升正式状态。
- 当前 formal active default 仍是 `subject_system_v1_governed_proactivity`。
- `ego_desktop_lab` 可以作为 lab / product-spine 实验面，但不覆盖 formal runtime state。
- 当前 autonomy ceiling 仍由 active lane 的 gate / evidence level 决定；未显式授权前，不自动升 higher gate。

## 用户固定输入

默认每一轮，用户只需要给 3 个控制输入：

1. `goal family`
   - `连续主体感`
   - `经历可塑性`
   - `proposal-only 主动提议`
   - `更强真实入口验证`
2. `risk ceiling`
   - `planning / synthetic only`
   - `bounded runtime internal changes`
   - `stronger live gate`
3. `gate policy`
   - `hold current tranche`
   - `allow next stronger bounded gate`

若用户未明确回答：

- 默认 `goal family = proposal-only 主动提议`
- 默认 `risk ceiling = bounded runtime internal changes`
- 默认 `gate policy = hold current tranche`

## Codex 固定动作

每一轮默认由 Codex 完成：

1. 把宽目标压成一个 `stage goal`
2. 把本轮内容压成一个唯一 `hypothesis`
3. 选一个最小 `change surface`
4. 先做最便宜的决定性验证
5. 只在当前 authority 允许层上改
6. 按需要跑三层验证：
   - `L1 local`
   - `L2 controlled`
   - `L3 live`
7. 明确写清：
   - 本轮证明了什么
   - 还没证明什么
   - 为什么不能写更强结论
8. 更新 task / campaign / evidence 记账

## 单轮固定闭环

每轮固定按这个顺序推进：

1. `Stage Card`
   - 本轮目标
   - 不证明什么
   - 本轮唯一 gate
   - 成功 / 失败判据
2. `One Hypothesis`
   - 只允许一个高价值假设
3. `One Change Surface`
   - 只选一个：
     - `memory / projection`
     - `self_state / writeback`
     - `proposal / governor`
     - `validation / artifact / scorer`
4. `Three-Level Verify`
   - `L1 local`
   - `L2 controlled`
   - `L3 live`
5. `Reviewer Verdict`
   - `success_reached`
   - `needs_more_implementation`
   - `needs_more_exploration`
   - `blocked_by_external_dependency`
   - `needs_reframing`
6. `Ledger Update`
   - progress checkpoint
   - experiment log
   - stage scorecard
   - task status / plan
   - 必要时 `PROGRAM_STATE_UNIFIED.yaml` 与 evidence ledger

## 什么时候必须停下来问用户

只有这些情况必须回到用户拍板：

- 两条高影响路线需要取舍
- 需要提升 claim ceiling 或升级 gate
- 需要外部依赖：
  - 账号
  - 审批
  - 真实入口
  - 人工观察
- 当前 framing 已被证明不值得继续
- 继续推进会明显扩大 scope 或改变产品方向

除此之外，默认不把实现细节抛回给用户。

## 每轮默认交付格式

每轮默认对用户只交付 4 段：

1. `本轮改了什么`
2. `本轮验证了什么`
3. `还没证明什么`
4. `下一轮唯一 frontier`

若没有低成本高收益的补充，明确写：

- `本轮无必要额外建议`

## 能力阶段顺序

未来若显式授权更高 gate，默认按这个顺序推进，不跳级：

1. `连续主体感`
2. `经历可塑性`
3. `proposal-only 主动提议`
4. `proposal 在真实入口稳定出现`
5. `更强跨入口稳定性`
6. 只有在以上都足够强时，才讨论更高 autonomy

在当前 frozen closeout 下，这个梯子只作为 future-stage discipline，不自动开启新 gate。
