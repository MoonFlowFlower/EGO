# Engineering Evidence Model

适用范围：repo 内需要判断问题层级、变更分类、验证等级、证据等级、完成口径的任务。

## 问题层级

按这个顺序判断当前问题所在层：

1. `目标`：真实目标、约束、验收口径不清
2. `策略`：主解决链、owner、入口、阶段不清
3. `表示`：信息组织错误，导致上下文漂移或读错 authority source
4. `实现`：代码、配置、接线、脚本需要改
5. `验证`：需要证明接入、启用、触发、生效
6. `收口`：已经有真实效果，正在沉淀、观察、归档、交接

## 变更分类

- `根因修复`
- `临时缓解`
- `旁路兜底`
- `观察增强`
- `架构变更`

如果不是根因修复，必须明确：

- 为什么主路当前不可直接修
- 引入了什么边界破坏或能力损失
- 回退条件是什么

## 三道门

### Gate A: Contract / Authority

先回答：

- authority source 是什么
- 真实目标和成功判据是什么
- 改的是哪一层问题
- 变更分类是什么

回答不清，不进入正式实现。

### Gate B: E2E / Mainline

改动必须能说明：

- 接到哪个真实入口
- 经过哪条主流程
- 改变哪个最小决策点
- 如何判断已接入 / 已启用 / 已触发 / 已生效

### Gate C: Preflight / Tool Doctor

交付前至少完成：

- 目标文件语法 / 导入有效
- 改动相关最小验证可执行
- 输出、日志、状态或 artifact 可用于验收

## 验证等级

- `V0`：未验证，仅方案
- `V1`：静态检查通过
- `V2`：局部测试或最小复现通过
- `V3`：子链路验证通过
- `V4`：主链真实触发证据通过
- `V5`：多样本或持续观察稳定通过

规则：

- 未到 `V4`，不得说“主线已闭环”
- 未到 `V5`，不得说“稳定解决”

## 证据等级

- `E0`：猜测
- `E1`：口头描述，未核验
- `E2`：局部代码 / 日志 / 文档片段
- `E3`：可重复局部验证
- `E4`：主链真实触发证据
- `E5`：持续观察或多样本稳定复现

规则：

- 结论强度不得高于证据强度
- 未到 `E4`，不得称“真实主线已生效”
- 未到 `E5`，不得称“稳定成立”

## Completion claim

- `可宣称完成`：当前任务目标已满足，证据和结论匹配
- `条件性完成`：实现或局部验证已完成，但主链证据或稳定性仍有缺口
- `不可宣称完成`：关键 authority、实现或验证仍未形成闭环

默认收口至少回答：

1. 改了什么
2. review 发现并修了什么
3. 实际跑了什么验证
4. 还没证明什么
